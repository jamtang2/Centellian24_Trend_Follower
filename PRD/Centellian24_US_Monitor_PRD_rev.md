# PRD: Centellian24_US_Monitor
### 센텔리안24(동국제약) 미국 시장 인기도 추적 대시보드

---

## 1. 배경 (Background)

- 동국제약의 화장품 수출액은 2024년 163억원 → 2025년 300억원 → 2026년 1,000억원 돌파 전망(LS증권 추정)이며, 특히 미국향 수출은 전년 대비 6~7배 성장이 기대되는 상황.
- 지난달(2026.06) 센텔리안24가 미국 뷰티 전문 유통 '얼타 뷰티(Ulta Beauty)' 1,400개 매장, '노드스트롬' 89개 매장에 입점 완료.
- 그러나 이는 **유통망 확장(공급 측 지표)** 일 뿐, 실제 미국 소비자들의 **수요/인기 반응(수요 측 지표)** 은 별도로 확인되지 않음.
- 회사 측 공식 실적(분기 IR, DART 공시)은 분기 단위로만 확인 가능해 반응 속도가 느림 → **주간 단위 선행지표(leading indicator)** 를 통해 미국 내 확산 추이를 조기에 포착하고자 함.
- 기존 구축 파이프라인(PRD → Claude Code → GitHub Actions → GitHub Pages, 예: Shortage_Bottleneck_Scanner, CAPEX_Dashboard)과 동일한 아키텍처를 재사용.
- **[아키텍처 변경 이력]** 초기 버전은 아마존/Google Trends 수집까지 GitHub Actions 러너에서 전부 실행하는 구조였으나, 시범 운영 중 GitHub Actions 러너의 데이터센터 IP가 아마존(503 오류)과 Google Trends(429 rate limit)에서 반복적으로 차단당하는 현상이 확인됨. 이는 시간이 지나도 해소되지 않는 IP 기반 봇 차단으로 판단되어, **아마존·Google Trends 수집은 사용자 로컬 PC(가정용 IP)에서 실행**하고, GitHub은 결과 데이터 저장·대시보드 배포 용도로만 사용하는 구조로 변경함(자세한 내용은 7장 참고).

## 2. 목적 (Goal)

센텔리안24의 미국 시장 내 소비자 관심도·판매 확산 정도를 나타내는 직·간접 지표를 **매주 1회 자동 수집**하고, **시계열 누적 대시보드**로 시각화하여 동국제약 화장품 사업 모멘�텀을 정성/정량적으로 조기 판단할 수 있게 한다.

| 4 | TikTok 해시태그 | #Centellian24, #MadecaCream 등 해시태그 게시물 수, 누적 조회수 | 간접(바이럴리티) | 높음 |

> 참고: 아마존/Ulta는 "판매량 지표"가 아니라 "판매 순위·리뷰 증가 속도"를 프록시로 사용하는 것이며, TikTok/Instagram 해시태그 지표는 공식 API 접근이 제한적이므로 수집 방식에 한계가 있음을 3장 하단 리스크에 명시.

## 4. 수집 방식 및 기술적 제약

### 4.1 아마존 — 직접 스크래핑 (사용자 로컬 PC에서 실행)
- 공식 Product Advertising API는 승인 절차·매출 실적 요건이 있어 개인 프로젝트로는 접근 제한 → **검색 결과 페이지 + 상품 상세 페이지 스크래핑**으로 진행(SerpApi 등 유료 서비스 미사용).
- **실행 위치: GitHub Actions 러너가 아닌 사용자의 로컬 PC**. GitHub Actions의 데이터센터 IP는 아마존이 봇으로 판별해 503으로 차단하는 빈도가 높아, 가정용 IP에서 실행해 정상적인 사용자 트래픽처럼 보이도록 함(7장 참고).
- 최소 수집 필드: 상품명, ASIN, **추정 월간 판매량**(상품 상세 페이지의 "지난달 N+ 구매" 배지 등에서 파싱), 카테고리명, 카테고리 내 BSR 순위, 리뷰 수, 평균 별점, 가격, 품절 여부.
- 대표 추적 상품 3~5개 선정(마데카 크림 타임 리버스, 360도 샷 PDRN 리프팅 아이크림, 마데카 크림 액티브 리뉴 PDRN 등 주력 라인 기준).
- 페이지 구조 변경으로 스크래핑이 실패하는 경우, 해당 주차는 이전 값을 유지하고 실패 로그만 남김(4.4의 grounding 폴백 대상에서는 제외 — 아마존은 스크래핑 전용 채널로 고정).

### 4.2 Google Trends — 미국 + 일본 (사용자 로컬 PC에서 실행)
- `pytrends`(비공식 라이브러리) 사용, 국가 코드 **US, JP 둘 다 조회**, 지난 12개월 롤링 윈도우로 조회 후 최신 주차 값만 누적 저장.
- **실행 위치: 아마존과 동일하게 사용자 로컬 PC**. 비공식 라이브러리 특성상 GitHub Actions의 공용 IP 대역에서는 429(rate limit)가 반복적으로 발생했기 때문.
- 키워드셋: "Centellian24", "Madeca Cream"(영문) + 필요 시 일본어 표기("センテリアン24" 등) 별도 트래킹 여부는 M2 단계에서 실데이터 확인 후 결정.
- 로컬 실행이라도 요청 빈도(주 1회)는 그대로 유지해 IP 차단 리스크를 최소화.

### 4.3~4.4 Ulta Beauty / TikTok / Instagram 해시태그 — Google Search grounding 전용 (사용자 로컬 PC에서 실행)
- 세 채널 모두 스크래핑을 시도하지 않고 **Gemini API + Google Search grounding**을 1차이자 유일한 수집 방식으로 사용.
- Gemini API 호출 자체는 IP 차단 이슈가 없어 GitHub Actions에서 유지해도 되지만, 아마존/Google Trends와 실행 시점을 동기화하고 구조를 단순하게 유지하기 위해 **하나의 로컬 스크립트(run_weekly.py)에서 5개 채널을 모두 순서대로 실행**하도록 통합.
- 질의 예시:
  - Ulta: "Ulta Beauty 웹사이트에서 Centellian24(센텔리안24) 제품의 현재 리뷰 수, 평균 별점, 입점 SKU 수, 품절 여부를 조사해줘"
  - TikTok: "TikTok에서 #Centellian24, #MadecaCream 해시태그의 게시물 수와 대표 영상 조회수를 조사해줘"
  - Instagram: "Instagram에서 #Centellian24 해시태그의 게시물 수를 조사해줘"
- 응답은 구조화된 JSON으로 반환하도록 프롬프트를 설계하고, 근거가 된 출처 URL을 함께 받아 저장.
- 이 방식으로 얻은 모든 수치는 **추정치(estimate)** 로 취급하며, 대시보드에 신뢰도 낮음 배지를 표기(아마존/Google Trends 대비 실측 정합성이 낮음을 명시).

### 4.5 큐텐 재팬(Qoo10 Japan) — 직접 스크래핑 (사용자 로컬 PC에서 실행)
- 공식 API 부재 → 아마존과 동일하게 **상품 상세 페이지 스크래핑**으로 진행, 실행 위치도 동일하게 **로컬 PC**(IP 차단 회피 목적, 7장 참고).
- 대표 제품 1~2개(마데카 크림 등 일본 판매 확인된 라인) 기준으로 수집.
- 수집 필드: 상품명, 판매 배지(구매수·베스트 순위 등 페이지에 노출되는 지표), 카테고리 내 순위, 리뷰 수, 평균 별점, 품절 여부.
- 페이지 구조(일본어 큐텐 재팬 사이트)가 한국 큐텐과 다를 수 있어 M10 단계에서 실제 페이지를 보며 셀렉터를 새로 확정해야 함.
- 스크래핑 실패 시 아마존과 동일하게 이전 값 유지 + 로그 기록.

### 4.6 아마존 리뷰 감성분석 — Gemini 텍스트 분석 (파생 지표)
- 새로운 원천 데이터 수집이 아니라, **4.1 아마존 스크래핑 단계에서 함께 가져온 리뷰 텍스트 상위 N개(예: 최신 10~20개)** 를 Gemini에 전달해 감성분석.
- 프롬프트 예시: "아래는 아마존에서 수집한 Centellian24 제품 리뷰 텍스트 목록이야. 전반적인 소비자 호감도를 0~100 점수로 매기고, 그렇게 판단한 근거를 2~3문장으로 요약해줘. 극단적으로 긍정적이거나 부정적인 리뷰가 소수 있어도 전체 평균 흐름을 기준으로 판단해줘."
- Google Search grounding은 필요 없음 (외부 검색이 아니라 이미 확보한 텍스트를 분석하는 작업이므로 grounding 없이 일반 Gemini 호출로 처리).
- 출력: `sentiment_score`(0~100), `sentiment_summary`(텍스트)를 아마존 제품별로 저장.
- 분석 대상 리뷰 수가 너무 적은 경우(예: 신규 제품, 리뷰 5개 미만) 신뢰도 낮음으로 표기.
- 저작권 고려: 리뷰 원문 전체를 그대로 저장/노출하지 않고, Gemini 분석 결과(점수·요약)만 데이터에 남김.

### 4.7 AI API 활용 (Google Gemini) — 역할 갱신
- 본 프로젝트는 **Gemini 단독 사용**. 기존 Claude/GPT/Gemini 앙상블 방식과 달리, Google Search grounding 기능이 핵심 수집 채널(4.3~4.4)이자 유일한 AI 호출 지점.
- 역할:
  1. Ulta / TikTok / Instagram 지표의 1차이자 유일한 수집 수단(grounding 검색)
  2. **아마존 리뷰 텍스트 감성분석**을 통한 호감도 점수 산출(grounding 불필요, 4.6 참고)
  3. 주간 수집 데이터 기반 정성적 요약 코멘트 생성("이번 주 아마존 추정 판매량 20,000개 유지, 아이크림 카테고리 1위 지속" 등)
  4. 출처 URL을 함께 반환하도록 프롬프트 설계(기존 프로젝트의 "출처 인용" 요구사항과 동일하게 적용)
- 모델: Gemini 최신 안정 버전(실행 시점 기준 확인 필요, grounding 기능 지원 모델 선택).
- **API 키 저장 위치**: 전체 수집이 로컬 실행으로 통합됨에 따라, GEMINI_API_KEY는 로컬 `.env` 파일(`.gitignore` 처리)에만 저장. GitHub Actions는 데이터 수집을 하지 않으므로 더 이상 GitHub Secrets에 이 키를 둘 필요가 없음(단, 기존에 등록해둔 Secret은 남겨두어도 무방하며, 혼동 방지를 위해 정리해도 됨).

## 6. 대시보드 구성 (GitHub Pages)

기존 프로젝트(Shortage_Bottleneck_Scanner 등)와 톤앤매너 통일.

1. **상단 요약 카드**: 최신 주차 기준 5개 지표 스냅샷 + WoW 변화율(▲▼ 색상 표기)
2. **누적 추이 차트 (핵심 화면)**
   - 아마존 제품별 추정 월간 판매량 추이(막대/선 그래프)
   - 아마존 BSR 순위 추이(역순 스케일, 낮을수록 상단)
   - 아마존/Ulta 리뷰 수 누적 추이(선 그래프)
   - Google Trends 지수 추이 — **미국/일본 두 라인 비교**
   - TikTok/Instagram 게시물 수·조회수 추이(추정치 배지 표시)
   - **아마존 vs Qoo10 재팬 판매 프록시 비교**(미국/일본 시장 반응 동시 비교 차트)
   - **아마존 리뷰 감성 점수(sentiment_score) 추이**(제품별 라인 그래프, 0~100 스케일)
3. **AI 주간 코멘트 섹션**: Gemini가 생성한 자연어 요약 + 참고 출처 링크
4. **원자료 테이블**: 주차별 raw 데이터 다운로드(CSV) 옵션
5. **신뢰도 표기**: 스크래핑 기반 실측(Amazon, Google Trends, Qoo10) vs Gemini grounding/분석 기반 추정(Ulta, TikTok, Instagram, 감성점수) 구분 뱃지

## 7. 자동화 아키텍처 (로컬 실행 + GitHub Pages 배포 분리)

IP 차단 문제로 인해 데이터 수집은 **사용자 로컬 PC**에서, 정적 사이트 배포는 **GitHub Pages**(기존 자동 배포 기능)에서 각각 처리하는 2단 구조로 변경.

```
[사용자 로컬 PC: Windows 작업 스케줄러 — 매주 일요일 06:17 KST 실행]
        │
        ├─ Step 1: Amazon 검색결과/상품 상세 페이지 스크래핑 (추정 판매량, BSR, 리뷰 수, 리뷰 텍스트 등)
        ├─ Step 2: pytrends 호출 — Google Trends US + JP
        ├─ Step 3: Qoo10 재팬 상품 상세 페이지 스크래핑 (판매 배지, 랭킹, 리뷰 수 등)
        ├─ Step 4: Gemini grounding 호출 — Ulta Beauty 리뷰/입점 현황
        ├─ Step 5: Gemini grounding 호출 — TikTok 해시태그
        ├─ Step 6: Gemini grounding 호출 — Instagram 해시태그
        ├─ Step 7: Gemini 호출(grounding 불필요) — 아마존 리뷰 텍스트 감성분석 → 호감도 점수 산출
        ├─ Step 8: 데이터 정합성 체크 + history 누적 저장
        ├─ Step 9: Gemini로 주간 코멘트 생성
        ├─ Step 10: 정적 대시보드(docs/index.html) 재생성
        └─ Step 11: git add/commit/push (자동 커밋 스크립트)
                │
                ▼
[GitHub: push 감지 → GitHub Pages가 /docs 폴더를 자동 재빌드·배포]
```

- **실행 위치 변경**: 기존엔 GitHub Actions 러너(데이터센터 IP)에서 Step 1~11을 전부 수행했으나, 아마존(503)·Google Trends(429) 차단이 반복 확인되어 **Step 1~11 전체를 로컬 PC 실행으로 이전**. GitHub Actions는 더 이상 데이터 수집 워크플로우(`weekly_update.yml`)를 실행하지 않으며, GitHub Pages의 기본 배포 트리거(push 시 자동 재빌드)만 그대로 활용.
- **실행 방식**: Windows 작업 스케줄러(Task Scheduler)에 매주 일요일 06:17 KST 트리거 등록 → `python -m scripts.run_weekly` 실행 → 스크립트 마지막 단계에서 git commit/push까지 자동 수행.
- **필수 전제 조건**: 트리거 시점에 **로컬 PC가 켜져 있고 인터넷에 연결되어 있어야** 자동 실행됨 (노트북이 꺼져있거나 절전 모드면 그 주는 실행되지 않음 — 9장 리스크 참고).
- git push에 필요한 자격증명(SSH 키 또는 `gh auth login` 토큰)이 로컬 PC에 미리 설정되어 있어야 함.
- 실패 시(스크래핑/Gemini 호출 등) 이전 주차 값 유지 + 로그 기록은 기존과 동일.
- 기존에 등록해둔 `.github/workflows/weekly_update.yml`(GitHub Actions용 주간 워크플로우)은 더 이상 사용하지 않으므로 리포지토리에서 제거하거나 비활성화 처리.

## 9. 리스크 및 한계

- **법적/약관 리스크**: 아마존은 자동 스크래핑을 이용약관으로 제한하는 경우가 있어, 상업적 배포가 아닌 **개인 리서치 목적의 저빈도(주 1회) 조회**로 한정하고 과도한 요청은 피해야 함.
- **데이터 신뢰도**: Ulta/TikTok/Instagram 수치는 Gemini 검색 grounding 결과에 의존하는 추정치로, 실제 값과 오차가 클 수 있음 → 절대값보다는 **추세(방향성)** 참고용으로 활용 권장. 아마존 "추정 판매량"도 아마존이 공식 제공하는 정확한 수치가 아니라 페이지 내 구매 배지 기반 추정값임에 유의.
- **소스 변경 취약성**: 아마존 페이지 HTML 구조 변경 시 스크래핑 실패 가능 → 정기적인 셀렉터 점검 필요.
- **아마존 카테고리 특정 어려움**: 마데카 크림 등 제품이 여러 카테고리에 걸쳐 있을 경우 BSR 비교 기준이 흔들릴 수 있음 → 추적 제품/카테고리를 명확히 고정.
- **Gemini grounding 응답 변동성**: 동일 질의라도 검색 결과 구성에 따라 매주 응답 형식/수치가 요동칠 수 있으므로, 프롬프트에 출력 스키마를 엄격히 지정하고 파싱 실패 시 재시도 로직 필요.
- **로컬 실행 의존성(신규)**: 자동화가 GitHub Actions가 아닌 로컬 PC의 작업 스케줄러에 의존하게 되면서, **PC가 꺼져 있거나 절전 모드거나 인터넷이 끊긴 경우 해당 주차 데이터 수집이 누락**됨. 완전 무인 자동화가 아니라는 점을 감안해, 실행 여부를 주기적으로 확인하거나(예: 실행 완료 시 알림 전송) 놓친 주차를 수동으로 재실행할 수 있는 절차를 마련 필요.
- **git 인증 만료 리스크(신규)**: 로컬 PC의 git 자격증명(토큰/SSH 키)이 만료되거나 갱신이 필요한 경우 자동 push가 조용히 실패할 수 있음 → 정기적으로 `gh auth status`로 인증 상태 점검 권장.
- **IP 차단이 로컬에서도 재발할 가능성**: 가정용 IP라 해도 지나치게 잦은 요청(예: 개발 중 반복 테스트)을 하면 일시적으로 차단될 수 있으므로, 로컬 개발/테스트 시에도 실행 빈도를 자제.
- **Qoo10 재팬 구조 불확실성(신규)**: 한국 큐텐과 일본 큐텐의 페이지 구조/노출 지표(판매 배지 문구 등)가 다를 수 있어, M10 구현 시 실제 페이지를 먼저 확인하고 셀렉터를 새로 잡아야 함. 판매량이 직접 노출되지 않고 "베스트 순위"만 보이는 경우, 아마존처럼 절대 판매량 대신 순위 변화만 지표로 사용할 수 있음.
- **감성분석의 표본 편향(신규)**: 아마존 리뷰는 만족/불만족이 극단적인 소비자가 남기는 경향이 있어(선택 편향), `sentiment_score`가 실제 평균 소비자 반응보다 다소 왜곡될 수 있음. 리뷰 수가 적은 신규 제품일수록 이 편향이 커지므로, 리뷰 수와 함께 점수를 해석하도록 대시보드에 안내 문구 추가 권장.
